■■■ HikariCP ■■■

○ HikariCP 개요 

	- HikariCP 라는 이름에서 Hikari 는 일본어로 빛을 의미한다

	- 이 이름은 일본인 개발자가
	  라이브러리의 핵심 가치인 빛처럼 빠른 속도를 강조하기 위해 붙인 것이다.
	  단순히 빠른 것을 넘어, 군더더기 없는 최적화를 통해
	  물리적으로 가능한 한계 속도에 도달하겠다는 의지가 담겨 있다.
	  (로고 역시 빛이 퍼져나가는 현상을 하고 있으며,
	   데이터베이스의 어플리케이션 사이의 병목 현상을 빛의 속도로 풀겠다는 상징성을 가진다.)

○ 설계적 특징 

	- 컬렉션 최적화
	  일반적인 ArrayList는 요소를 제거할 때 처음부터 끝까지 스캔하지만
	  HikariCP는 FastList라는 커스텀 리스트를 사용하여 역순으로 스캔한다.
	  커넥션 풀의 특성상 최근에 사용한 리소스를 먼저 해제하는 경우가 많기 때문에
	  이 미세한 차이가 성능 향상으로 이어진다.

	- 바이트코드 수준의 슬림한
	  자바 컴파일러가 생성하는 바이트코드를 분석하여,
  	  CPU 캐시 히트율을 높이고 명령어를 최소화하도록 코드를 작성했다.
	  내부 구조는 다이어트 코드 상태이다.

	- 오버헤드 제로를 목표로 하는 극한의 성능 최적화로
	  현존하는 자바 커넥션 풀 라이브러리 중 가장 빠른 응답 속도를 구현


○ 스프링에서 HikariCP를 핵심적으로 사용하는 이유

	- 스프링 부트 2.0 버전부터 기본 커넥션 풀로 채택된 HikariCP는
	  제로 오버헤드를 자랑하는 매우 빠르고 가벼운 JDBC 커넥션 풀 라이브러리

	- 압도적인 성능
	  HikariCP는 다른 커넥션 풀(DBCP, C3P0, Tomcat JDBC 등)과 비교했을 때
	  벤치마크 결과가 월등히 좋다.
	- FastPath Pooling
	  → 커넥션을 얻고 반납하는 과정에서 불필요한 동기화를 줄여 지연 시간을 최소화했다. 
	  바이트코드 최적화
	  → 컴파일된 바이트코드 수준까지 최적화하여 CPU 오버헤드를 줄였다.

	- 경량화 및 리소스 효율성 
	  작은 용량
	  → 라이브러리 크기가 약 1300~160kb 정도로 매우 작아 애플리케이션 메모리 부담이 적다.
	   = Proxy객체 최적화 
	  → Statement나 ResultSet 등을 처리할 때 사용하는 프록시 객체의 성능을 최소화하여
	     가비지 컬렉션의 부하를 줄였다.

	- 안정성과 신뢰성
	  → 장애 감지 및 복구 
	     = 네트워크 문제나 DB 서버 이슈로 커넥션이 끊어졌을 때
	       이를 감지하고 복구하는 로직이 매우 강력하다.

	- 데드락 방지
	  → 커넥션 풀의 크기를 관리하는 알고리즘이 잘 설계되어 있다.
	     고부하 상황에서도 시스템이 멈추지 않도록 돕는다.

	- 간결한 설정
	  → 복잡한 설정 없이 기본값만으로도 충분히 훌륭한 성능을 내도록 설계되어 있다.
	     개발자는 spring?-pool-size나 connection-timeout같은
	     핵심적인 옵션 몇 가지만 조정하여 사용하면 된다.

※ 커넥션 풀 작동 원리
   커넥션 풀은 매번 DB 연결을 맺고 끊는 비용을 줄이기 위해
   이와 연결된 객체들은 풀에 담아두고 재사용하는 방식
    ·Connection Reuse → 미리 생성된 커넥션을 재사용하여 TCP 핸드셰이크 시간을 절약
    ·Throtting → 동시에 가동되는 커넥션 수를 제한하여 DB 서버의 부하를 방지
    ·Validation → 사용자가 전 커넥션이 유효한지 체크하여 에러를 사전에 방지
